# 실시간 권한 변경 처리 (IEUM-90) 작업 요약

## 1. 개요 및 배경 (왜 했는지)
MarkFlow 프로젝트에서 **소유자가 다른 멤버의 권한(편집자 ↔ 뷰어)을 변경하거나 추방했을 때**, 변경 대상이 된 멤버의 화면이 즉시 업데이트되지 않고 미아 상태가 되거나 과거 권한 상태를 유지하는 버그가 있었습니다.
이로 인해 뷰어로 강등된 사용자가 편집을 시도하다가 서버에서 `HTTP 403 (Forbidden)` 에러를 마주치거나, 반대로 편집자로 승급되었음에도 불구하고 새로고침을 하기 전까지는 편집 도구를 사용할 수 없는 불편함이 있었습니다.

## 2. 원인 분석 및 해결 과정 (어떻게 했는지)

### 문제 상황 1: 뷰어로 강등 시 403 에러 발생 및 UI 불일치
백엔드(`CanvasGateway`)는 권한이 변경되면 즉각 보안을 위해 해당 유저의 웹소켓 연결을 끊습니다(`s.disconnect(true)`). 
그러나 프론트엔드는 소켓이 끊겼음을 감지하고도 아무런 조치를 취하지 않아, UI는 여전히 '편집자' 상태로 남아있었습니다.

#### 💡 해결 방안
`apps/web/src/collab/useSocketCollab.ts`에 존재하던 주석 처리된 로직을 복구 및 개선했습니다.
서버에 의해 웹소켓이 강제 종료된 경우(`reason === "io server disconnect"`), 즉시 브라우저 알림(alert)을 띄우고 페이지를 강제 새로고침(`window.location.reload()`)하도록 변경했습니다.
이를 통해 뷰어로 강등될 경우 0.1초 만에 최신 권한 상태(뷰어)로 화면이 다시 그려지며 403 에러를 원천 차단했습니다.

---

### 문제 상황 2: 뷰어 상태에서 다른 사람의 실시간 작업 미노출 및 승급 버그
유저님의 핵심 요구사항 중 하나는 **"뷰어도 남들이 움직이는 건 실시간으로 볼 수 있어야 한다"**였습니다. 
하지만 기존 프론트엔드 코드(`ProjectCollabLayout.tsx`)에는 리소스를 아끼겠다는 목적으로 **'뷰어 권한을 가진 유저는 아예 웹소켓 접속을 막는 하드코딩'**이 되어 있었습니다.

이로 인해 두 가지 치명적인 문제가 있었습니다.
1. 소켓이 없으니 뷰어는 남들의 실시간 이동/편집을 구경할 수 없었음.
2. 소켓에 아예 없으니, 소유자가 뷰어를 편집자로 승급시켜주며 날리는 "소켓 강제 종료 신호"조차 받지 못해 새로고침(알림)이 발생하지 않았음.

#### 💡 해결 방안
`apps/web/src/features/canvas/ProjectCollabLayout.tsx`에서 뷰어의 소켓 접속을 막던 차단 로직(`|| role === "VIEWER"`)을 삭제했습니다.
뷰어도 당당하게 소켓 방에 접속하여 실시간 이벤트를 수신할 수 있게 되었고, 편집자로 승급될 때도 정상적으로 서버의 신호를 받아 즉시 화면이 전환되게 되었습니다.
(※ 소켓에 접속하더라도 화면 UI 비활성화 및 백엔드의 `assertEditorPermission` 가드가 있어 무단 편집은 완벽히 방어됩니다.)

## 3. 요약 (무엇을 어떻게 고쳤는지)
1. **`apps/web/src/collab/useSocketCollab.ts`**
   - 서버 강제 종료 시 `alert("프로젝트 권한이 변경되어 새로고침합니다."); window.location.reload();` 실행 추가.
2. **`apps/web/src/features/canvas/ProjectCollabLayout.tsx`**
   - 뷰어 권한 유저의 소켓 접속 차단 코드 제거 (`if (role === undefined || role === "VIEWER") return;` ➡️ `if (role === undefined) return;`)
3. **`apps/web/src/store/canvasStore.ts`** (보너스 수정)
   - React Flow 버전 호환성 문제로 발생하던 `EdgeChange` 관련 TypeScript 빌드 에러를 Type Guard 방식으로 픽스.

# 프로젝트 삭제 시 강제 퇴장 처리 (IEUM-91) 작업 요약

## 1. 개요 및 배경 (왜 했는지)
A(소유자)가 B(편집자)가 접속 중인 상태에서 **프로젝트를 삭제했을 때**, B가 캔버스에서 강제 퇴장되지 않고 그대로 남아있는 버그가 발견되었습니다. 또한 기존 백엔드의 권한 검사 로직(`assertPermission`)이 캔버스 조회 시 프로젝트 삭제 여부보다 "멤버인가?"를 먼저 검사하는 바람에 404가 아닌 403 에러를 뱉어서 프론트엔드가 적절하게 대응하지 못하는 이슈도 있었습니다.

## 2. 해결 방안 (어떻게 했는지)
1. **백엔드 (API)**: 
   - `ProjectService`에서 프로젝트 삭제 시 사내 이벤트 버스를 통해 `PROJECT_DELETED` 이벤트를 쏘도록 추가했습니다.
   - `CanvasGateway`가 해당 이벤트를 수신하면, 권한이 강등됐을 때(`MEMBER_ROLE_CHANGED`)와 100% 동일하게 해당 캔버스 방의 모든 접속자의 **소켓을 즉시 강제 종료(`s.disconnect(true)`)** 하도록 구현했습니다.
   - `permission.ts` 파일 내의 `assertPermission` 함수에서, 멤버 정보를 찾을 수 없을 때 프로젝트 자체가 삭제된 것인지 추가 검사하여 삭제된 프로젝트라면 403이 아닌 **404 (Not Found)** 에러를 던지도록 고쳤습니다.
2. **프론트엔드 (Web)**: 
   - 서버가 강제로 소켓을 끊으면 프론트엔드는 권한 변경으로 간주하여 `window.location.reload()`(새로고침)를 실행합니다.
   - 새로고침 되면서 캔버스를 로드할 때 백엔드가 404 에러(프로젝트 삭제됨) 혹은 403 에러(강퇴됨)를 반환하면 프론트엔드의 `catch` 블록이 이를 잡아냅니다.
   - 에러 상태코드에 맞게 "프로젝트가 삭제되어 프로젝트 목록으로 이동합니다." 또는 "프로젝트 접근 권한이 없어 목록으로 이동합니다." 라는 팝업을 띄우고 즉시 **`/projects` 화면으로 강제 이동(리다이렉트)** 시키는 로직을 추가했습니다.

## 3. 요약 (무엇을 고쳤는지)
1. **`apps/api/src/common/events/project-events.service.ts`**: `PROJECT_DELETED` 이벤트 타입 추가
2. **`apps/api/src/modules/projects/project.service.ts`**: 삭제 시 이벤트 방출 로직 추가
3. **`apps/api/src/realtime/canvas.gateway.ts`**: `PROJECT_DELETED` 발생 시 해당 방 소켓 전체 강제 종료
4. **`apps/api/src/shared/permission.ts`**: 프로젝트 삭제 시 403 대신 404 에러 반환 로직 보강
5. **`apps/web/src/features/canvas/index.tsx`**: 404/403 에러 시 `/projects` 리다이렉트 로직 보강
