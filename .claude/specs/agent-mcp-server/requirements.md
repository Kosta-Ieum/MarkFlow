# AI 에이전트용 MCP 서버 — 요구사항

## 1. Overview

### 문제 정의
AI 에이전트(Claude Code 등)가 MarkFlow 캔버스를 읽고 편집할 표준 통로가 없다. REST·소켓 계약은 기계 판독형 정본(openapi.yaml, packages/shared zod)으로 이미 존재하지만, 에이전트가 이를 도구(tool)로 쓰려면 MCP(Model Context Protocol — AI 에이전트가 외부 시스템을 도구로 호출하는 표준 프로토콜) 서버 어댑터가 필요하다.

### 목표
로컬 stdio MCP 서버(`apps/mcp`)를 만들어, 에이전트가 **봇 계정으로** MarkFlow 프로젝트를 조회하고 캔버스(노드·엣지)를 편집할 수 있게 한다. 에이전트의 편집은 **기존 협업 경로(소켓)를 그대로 타서** 다른 참가자 화면에 실시간 반영된다.

### Non-goals (이번 범위 제외)
- 원격 HTTP 배포·OAuth·PAT — 로컬 stdio만. 단 전송 계층은 후일 HTTP 전환이 쉽게 분리(인터뷰 확정 ④).
- 채팅 송수신, 멤버 관리, 프로젝트 생성·수정·삭제, 영구삭제(휴지통 비우기).
- 커서 위치 emit(에이전트 커서 표시), 소프트 락 획득.
- BE(apps/api)·packages/shared·openapi 변경 — 어댑터는 기존 계약만 소비한다.
- undo/redo 히스토리 연동(에이전트 동작은 각 참가자의 per-user 스택 대상이 아님 — 기존 설계 그대로).

### 성공 기준
- Claude Code에 MCP 서버로 등록 후, 에이전트가 "프로젝트 목록 → 캔버스 읽기 → 노드 여러 개 생성 + 엣지 연결" 시나리오를 완주한다.
- 그 편집이 브라우저로 접속한 사용자 화면에 새로고침 없이 실시간 반영되고, 봇이 접속자 목록에 보인다.
- 서버 재시작 없이 토큰 만료를 스스로 복구한다.

### 운영 전제 (1회 수동 셋업)
- 봇 계정은 Gmail `+` 별칭 등으로 **1회 수동 가입**(이메일 인증 포함)하고, 편집할 프로젝트에 **EDITOR 멤버로 초대**되어 있어야 한다.
- 단일 세션 정책상 같은 봇 계정의 MCP 서버 인스턴스는 동시에 하나만 안정 동작(중복 실행 시 재로그인 경합) — 문서로 명시한다.

## 2. 요구사항

### R1. 서버 기동과 도구 노출 (stdio)

**유저스토리:** 에이전트 사용자로서, MCP 클라이언트(Claude Code 등)에 서버를 등록하면 MarkFlow 도구들이 보이길 원한다.

**인수 조건:**
- R1.1 WHEN MCP 클라이언트가 stdio로 서버를 기동하면, THE SYSTEM SHALL 정의된 도구 목록(R3·R4)을 스키마와 함께 노출한다.
- R1.2 THE SYSTEM SHALL 대상 서버 주소(API·WS)와 봇 자격증명을 환경변수로 받는다(코드에 하드코딩 금지).
- R1.3 IF 필수 환경변수가 없으면, THEN THE SYSTEM SHALL 기동 시(또는 최초 도구 호출 시) 어떤 변수가 없는지 명확한 에러로 알린다.

### R2. 봇 인증 수명주기

**유저스토리:** 운영자로서, 봇이 스스로 로그인·토큰 갱신을 처리해 서버를 방치해도 계속 동작하길 원한다.

**인수 조건:**
- R2.1 WHEN 최초 도구 호출이 발생하면, THE SYSTEM SHALL env 자격증명으로 로그인해 access token을 확보한다.
- R2.2 IF API가 401을 반환하면, THEN THE SYSTEM SHALL 재로그인(또는 refresh) 후 원 요청을 1회 재시도한다.
- R2.3 IF 자격증명이 틀리면, THEN THE SYSTEM SHALL 도구 결과로 인증 실패 원인을 반환한다(무한 재시도 금지).
- R2.4 THE SYSTEM SHALL 자격증명·토큰을 로그·도구 결과에 노출하지 않는다.

### R3. 읽기 도구

**유저스토리:** 에이전트로서, 편집 전에 현재 상태를 파악하기 위해 프로젝트·캔버스·히스토리를 조회하고 싶다.

**인수 조건:**
- R3.1 THE SYSTEM SHALL 봇이 멤버인 프로젝트 목록 조회 도구를 제공한다(id·이름·역할 포함).
- R3.2 THE SYSTEM SHALL 캔버스 스냅샷 조회 도구를 제공한다(노드 전체 필드 + 엣지 + 봇의 role).
- R3.3 THE SYSTEM SHALL 프로젝트 활동 히스토리 조회 도구를 제공한다(페이지네이션 포함).
- R3.4 THE SYSTEM SHALL 휴지통 목록 조회 도구를 제공한다(복원 대상 탐색용).

### R4. 캔버스 편집 도구

**유저스토리:** 에이전트로서, 아이디어 맵을 만들기 위해 노드와 엣지를 생성·수정·삭제·복원하고 싶다.

**인수 조건:**
- R4.1 THE SYSTEM SHALL 노드 생성 도구를 제공한다(title·markdown·type·position 지정 가능, 미지정 시 합리적 기본값).
- R4.2 THE SYSTEM SHALL 노드 수정 도구를 제공한다(title·markdown·type·position·collapsed 부분 수정).
- R4.3 THE SYSTEM SHALL 노드 삭제(휴지통 이동)·복원 도구를 제공한다.
- R4.4 THE SYSTEM SHALL 엣지 연결·해제 도구를 제공한다(중복 연결·self-loop은 서버 거부를 그대로 전달).
- R4.5 THE SYSTEM SHALL 모든 도구 입력을 packages/shared의 zod 스키마(재사용)로 검증하고, 위반 시 어떤 필드가 왜 틀렸는지 반환한다.

### R5. 실시간 협업 정합

**유저스토리:** 브라우저 참가자로서, 에이전트의 편집이 동료의 편집처럼 즉시 내 화면에 보이길 원한다.

**인수 조건:**
- R5.1 WHEN 에이전트가 편집 도구를 실행하면, THE SYSTEM SHALL 변경을 기존 협업 경로(FE와 동일한 소켓 이벤트 또는 브로드캐스트되는 REST)로 반영해 다른 참가자 화면에 실시간 표시되게 한다.
- R5.2 WHEN 봇이 프로젝트 룸에 접속하면, THE SYSTEM SHALL 접속자 목록(presence)에 봇 계정이 표시되게 한다(기존 presence 메커니즘 그대로).
- R5.3 IF 소켓이 끊기면, THEN THE SYSTEM SHALL 다음 편집 도구 호출 시 재접속을 시도하고, 불가하면 실패 원인을 반환한다.
- R5.4 THE SYSTEM SHALL 다른 참가자가 소프트 락으로 편집 중인 노드에 대한 수정·삭제 실패(서버 거부 포함)를 에이전트에게 명확히 전달한다.

### R6. 에러 처리 (에이전트 가독성)

**유저스토리:** 에이전트로서, 실패했을 때 다음 행동을 판단할 수 있는 에러를 받고 싶다.

**인수 조건:**
- R6.1 IF 봇의 role이 VIEWER(또는 비멤버)라 편집이 거부되면, THEN THE SYSTEM SHALL 권한 부족과 필요한 역할을 도구 결과로 반환한다.
- R6.2 IF 대상 노드·엣지·프로젝트가 없으면, THEN THE SYSTEM SHALL NOT_FOUND와 대상 id를 반환한다.
- R6.3 IF 서버에 연결할 수 없으면, THEN THE SYSTEM SHALL 네트워크 오류임을 구분해 반환한다(인증 오류와 혼동 금지).
- R6.4 THE SYSTEM SHALL 도구 실패를 MCP 에러 규약(isError)으로 반환하고 서버 프로세스는 죽지 않는다.

### R7. 워크플로 제약
- R7.1 구현이 완성되어도 **사용자 승인 전까지 PR을 생성하지 않는다** (사용자 지시 — 브랜치 커밋까지만).
