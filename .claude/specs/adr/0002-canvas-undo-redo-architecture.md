# ADR-0002: 캔버스 Undo/Redo 아키텍처 — 클라이언트 per-user 커맨드 스택

- **날짜**: 2026-07-15
- **상태**: 승인됨
- **관련 기능**: canvas-undo-redo

## 맥락
실시간 협업 캔버스에 undo/redo가 필요하다. 캔버스 상태는 **서버에 저장되고 모든 참가자가 공유하며 낙관적(optimistic)으로 갱신**된다(로컬 먼저 반영 → 소켓 전파 → 서버 저장). 이 위에 되돌리기를 얹는 방법은 여러 가지고, 고른 방식에 따라 서버 작업량·복잡도·일정이 크게 갈린다. 요구사항은 per-user(내 동작만), 세션 메모리, 대상 6종(노드 생성·삭제·이동, 엣지 연결·해제, 제목·내용 수정)이다.

## 검토한 대안
1. **클라이언트 커맨드 스택 + 기존 `applyLocal*` 역연산 재적용** — 각 클라이언트가 "자기가 한 동작"을 `{되돌리기, 다시하기}` 한 쌍(커맨드)으로 로컬 스택에 쌓고, undo는 그 역연산을 **기존 편집 경로(소켓 emit + 서버 저장 + 히스토리 로그)로 다시 실행**한다.
   - 장점: 새 서버 API·DB 없음(BE 변경 0), 기존 협업 배관 재사용, per-user가 자연스러움(각자 로컬 스택), 구현·리스크 최소.
   - 단점: 이동/내용의 "이전 값"을 코드가 안 남기므로 커맨드 기록 시 직접 캡처해야 함. 서버가 변경을 거부해도 자동 롤백 경로가 없음(기존 앱에도 없음). 새로고침 시 소실.
2. **서버 사이드 연산 로그 + 공유/글로벌 undo** — ActivityLog 확장 또는 전용 테이블에 연산을 적재하고, 서버가 되돌리기를 조율.
   - 장점: 영속·글로벌 undo·재접속 후 유지 가능.
   - 단점: 서버 작업이 큼(순서 합의·동시성), 4주 MVP엔 과함. per-user를 서버에서 다루는 추가 복잡도.
3. **CRDT/OT 기반 통합 편집 히스토리** — 문서/캔버스를 CRDT로 재모델링.
   - 장점: 동시편집·되돌리기를 이론적으로 정확히 처리.
   - 단점: 대형 라이브러리 도입 + 상태 모델 전면 재설계. 범위·일정 초과.

## 결정
**대안 1**(클라이언트 per-user 커맨드 스택, 역연산 재적용)을 채택한다. undo/redo 스택은 세션 메모리(Zustand)에 두고, undo/redo 실행은 반드시 `applyLocal*`(emit 포함) 계열을 재호출해 다른 참가자·서버에 동일하게 반영한다. BE·`packages/shared`·openapi 변경은 없다(apps/web 국한).

## 결과
- **BE 무변경** — 되돌리기는 기존 연산(add/delete/restore/move/edge/update)을 역방향으로 재사용할 뿐이다. 새 계약·마이그레이션 없음.
- **이전 값 캡처 책임이 프론트에** 생긴다 — 각 연산 실행 지점에서 이동 전 좌표·수정 전 제목/내용·삭제되는 엣지를 커맨드에 담아야 한다.
- **서버 거부 완전 롤백은 미지원**(best-effort) — 현재 REST는 fire-and-forget이고 클라이언트가 재동기화를 요청하는 경로가 없다. 낙관적 모델 위의 되돌리기이며, 서버 강제 재동기화(`sync:resync` 수신) 시 스택이 실제 상태와 어긋날 수 있어 그 경우 스택을 무효화한다.
- **영속·글로벌 undo는 후속 과제** — 필요해지면 대안 2로 확장(이 ADR을 대체하는 새 ADR).
