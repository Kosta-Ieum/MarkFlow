# ADR-0003: 에이전트 접속 아키텍처 — 봇 계정 + FE 동일 경로(소켓 하이브리드) MCP 어댑터

- **날짜**: 2026-07-20
- **상태**: 승인됨
- **관련 기능**: agent-mcp-server

## 맥락
AI 에이전트가 MarkFlow를 도구로 쓰려면 MCP 서버가 필요하다. 서버가 MarkFlow에 "어떤 자격으로, 어떤 경로로" 접속하는지가 핵심 결정이다. 제약: BE는 이메일 가입 + 단일 세션(재로그인 시 기존 refresh 무효화 + 소켓 강제 종료), refresh는 httpOnly 쿠키 전용, 노드 생성·수정은 FE가 소켓으로 직접 전파(REST 쓰기엔 브로드캐스트 없음), 4주 MVP 일정.

## 검토한 대안
1. **봇 계정 + FE 동일 경로 어댑터** — env 자격증명으로 로그인, 읽기는 REST, 편집은 FE와 같은 소켓 emit(노드 생성·수정·엣지) + 브로드캐스트되는 REST(삭제·복원).
   - 장점: BE·shared·openapi 변경 0. 검증된 브로드캐스트·권한 재검사·presence를 그대로 얻어 에이전트가 "진짜 협업자"로 보임. 지금 바로 구현 가능.
   - 단점: 봇 계정 1회 수동 가입 필요. 같은 봇 계정 동시 인스턴스 불가(단일 세션). Set-Cookie 수동 파싱. FE 송신 경로가 바뀌면 어댑터도 따라가야 함.
2. **REST-only 어댑터** — 소켓 없이 REST만.
   - 장점: 가장 단순.
   - 단점: 노드 생성·수정이 다른 참가자 화면에 실시간 반영 안 됨(새로고침 필요) — 협업 캔버스의 핵심 가치 훼손.
3. **BE 확장(PAT + REST 브로드캐스트 보강) 후 어댑터** — API 키 인증 추가, REST 쓰기에 소켓 브로드캐스트 부착.
   - 장점: 구조적 정석. 어댑터가 단순해지고 다중 인스턴스·원격 배포에 유리.
   - 단점: BE 도메인·공개 계약 변경 → 팀 합의·일정 필요. MCP 작업이 BE 작업에 종속.

## 결정
**대안 1**을 채택한다. 로컬 stdio MVP 기준 최단 경로이며 실시간 협업 정합을 잃지 않는다. 전송 계층은 분리해 두어 후일 원격(HTTP)·PAT 전환(대안 3 방향) 시 어댑터 재작성 없이 확장한다.

## 결과
- **BE 무변경** — 어댑터는 기존 계약(openapi + shared zod + 소켓 이벤트)만 소비한다.
- **운영 전제가 생김** — 봇 전용 계정 1회 가입 + 프로젝트 EDITOR 초대, 동시 인스턴스 1개 제한(문서화).
- **FE 송신 경로와의 결합** — FE가 연산별 송신 경로(소켓 vs REST)를 바꾸면 어댑터도 동기화해야 한다. 정본은 계속 openapi·shared이므로 스키마 차원 드리프트는 없음.
- **후속 확장 경로** — 원격 배포·다중 인스턴스가 필요해지면 PAT + REST 브로드캐스트 보강(대안 3)으로 이 ADR을 대체하는 새 ADR을 쓴다.
